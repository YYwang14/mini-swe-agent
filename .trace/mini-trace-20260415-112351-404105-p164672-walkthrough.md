## Fundamental Architecture

The run has three main runtime objects:

| Object | Created at trace lines | Main source | Role |
|---|---:|---|---|
| Model | `74-121` | `src/minisweagent/models/litellm_model.py` | Calls the LLM through LiteLLM and parses tool calls. |
| Environment | `122-133` | `src/minisweagent/environments/local.py` | Executes bash commands locally with `subprocess.run`. |
| Agent | `134-155` | `src/minisweagent/agents/interactive.py` and `src/minisweagent/agents/default.py` | Owns the loop, messages, execution, and saving. |

The core loop is:

```text
InteractiveAgent.step()
  -> DefaultAgent.step()
    -> query()
      -> LitellmModel.query()
        -> litellm.completion(...)
        -> parse_toolcall_actions(...)
    -> execute_actions(...)
      -> LocalEnvironment.execute(...)
      -> format_observation_messages(...)
```

The important design point: the LLM does not directly edit files. It returns structured bash tool calls. The Python agent parses those calls, executes them through the local environment, then appends command output back into the message history.

## Expected Trace Artifacts

### `JSONDecodeError` Events

Some `exception` rows in the trace are `JSONDecodeError`. These usually come from message display/formatting code that tries to parse message content as JSON.

That happens because observations are JSON strings, but ordinary assistant text is not. The formatter attempts JSON parsing, catches failures, and falls back to plain text.

These events are expected and are not task failures.

### `Submitted` Exceptions

`Submitted` exceptions are intentional. They are how mini-swe-agent exits the loop after the model emits the required completion command:

```bash
echo COMPLETE_TASK_AND_SUBMIT_FINAL_OUTPUT
```

Relevant trace lines:

| Trace line | Meaning |
|---:|---|
| `751` | `_check_finished()` checks the command output. |
| `754` | `_check_finished()` raises `Submitted`. |
| `756` | `LocalEnvironment.execute()` propagates `Submitted`. |
| `758` | `InteractiveAgent.execute_actions()` sees `Submitted`. |
| `802` | `DefaultAgent.run()` sees `Submitted`. |
| `872` | `DefaultAgent.run()` returns final status `Submitted`. |

### Final `action was not executed` Observation

Trace line `783` records an observation containing:

```text
action was not executed
```

That does not mean the submit command failed. The submit command did run. The reason this appears is:

1. `LocalEnvironment.execute()` runs the submit command.
2. `_check_finished()` detects the sentinel output.
3. `_check_finished()` raises `Submitted` before the normal command output is appended to the outputs list.
4. The observation formatter runs in a `finally` path.
5. Since the output list has no entry for that action, the formatter pads a placeholder.

So this line is a tracing/control-flow artifact, not a failed command.

Agent: keeps conversation state and decides when to call model or environment
Model: receives messages and returns bash tool calls
Environment: executes bash commands and returns observations
